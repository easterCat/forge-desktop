## 2. UI 整改清单逐项评估

### 2.1 评估框架

针对 UI 整改清单中的每项问题，从以下维度评估后端介入必要性：

1. **是否需要持久化？** → 如需跨设备/会话同步，需后端存储
2. **是否涉及敏感数据？** → 如涉及 Token 等敏感信息，需加密存储
3. **是否有性能考量？** → 如需大量计算，需后端处理
4. **是否有安全考量？** → 如涉及访问控制，需后端鉴权

### 2.2 评估结果

#### 2.2.1 Design Token 管理

| 问题 | 评估结果 |
|------|---------|
| **UI 描述** | 建立统一的 Design Token API |
| **后端介入必要性** | ❌ 不需要 |
| **原因** | Design Token 为静态配置，无需后端 API 提供 |
| **现有方案** | `src/assets/theme.css` 已包含所有 Design Token（颜色、间距、字体等） |
| **前端实现** | CSS 变量（`--accent`、`--bg-card` 等）直接引用 |

**结论**：Design Token 以静态 CSS 文件形式存在，前端直接引用，无动态获取需求。

#### 2.2.2 主题配置管理

| 问题 | 评估结果 |
|------|---------|
| **UI 描述** | 多主题系统可能需要主题配置管理接口 |
| **后端介入必要性** | ❌ 不需要 |
| **原因** | 主题配置为用户本地偏好，无需跨设备同步 |
| **现有方案** | `src/composables/useTheme.ts` 已实现完整主题管理 |
| **存储方式** | `localStorage.setItem(STORAGE_KEY, theme)` |
| **支持主题** | light / dark / warm / glass / yellow |

**结论**：主题配置完全由前端 localStorage 管理，无后端介入必要。如未来需要跨设备同步主题偏好，可考虑增加后端支持，但当前需求不在范围内。

#### 2.2.3 用户偏好设置

| 问题 | 评估结果 |
|------|---------|
| **UI 描述** | 主题选择等用户设置可能需要持久化 |
| **后端介入必要性** | ❌ 不需要 |
| **原因** | 用户偏好为本地配置，无共享/同步需求 |
| **现有方案** | 主题偏好存储于 localStorage |
| **持久化机制** | 页面刷新后保持用户选择 |

**结论**：用户偏好设置（主题、语言、视图模式等）均可在前端独立实现，无需后端数据库支持。

### 2.3 纯前端主导问题（无需后端介入）

以下 UI 整改问题属于纯前端实现范畴：

| 问题类别 | 描述 | 前端实现方式 |
|---------|------|-------------|
| **CSS 样式规范化** | 统一按钮、输入框等组件样式 | `theme.css` CSS 变量覆盖 |
| **按钮组件统一** | 统一主/次/幽灵按钮样式 | `.btn-primary` / `.btn-secondary` / `.btn-ghost` |
| **响应式布局完善** | 多断点适配 | Tailwind CSS / 原生媒体查询 |
| **无障碍标签** | 添加 aria-label 等 | Vue 组件属性添加 |
| **图标系统** | 导航/操作/状态图标规范 | `design/icons/` SVG 文件 |

## 3. 现有架构评估

### 3.1 当前数据流架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Vue 3 前端（Pinia + localStorage）           │
├─────────────────────────────────────────────────────────────┤
│  useTheme.ts                                                 │
│  ├── currentTheme: ref<Theme>                               │
│  ├── setTheme(): applyThemeToDOM() + localStorage.setItem() │
│  └── cycleTheme(): 循环切换 5 个主题                         │
├─────────────────────────────────────────────────────────────┤
│                    Tauri IPC Bridge                           │
├─────────────────────────────────────────────────────────────┤
│                    Rust 后端（SQLite）                         │
│  └── 用于：插件/技能/MCP/备份 等业务数据持久化                 │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 主题管理的边界划分

| 层级 | 职责 | 说明 |
|------|------|------|
| **CSS 变量层** | 定义 Design Token | `theme.css` 中的 `--*` 变量 |
| **应用逻辑层** | 主题切换逻辑 | `useTheme.ts` 中的 `setTheme()` |
| **持久化层** | 用户偏好存储 | `localStorage` |
| **业务数据层** | 业务数据持久化 | SQLite（后端） |

当前架构中，用户偏好设置（主题选择）位于 **持久化层**，由前端 localStorage 负责，符合本地优先（local-first）设计原则。
